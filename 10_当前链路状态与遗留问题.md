# 当前链路状态与遗留问题

## 1. 当前已打通的完整链路

当前 CreatorPulse MVP 已经形成一条端到端数据链路：

```text
用户注册
-> 创建 user
-> 创建 creator_id
-> 选择模拟绑定平台
-> 初始化该达人基础数据
-> mock_generator 按 MySQL 绑定配置生成事件
-> Flume 监听文件并推送到 Kafka
-> Kafka 保存 video_stats_topic 消息
-> Spark Streaming 消费 Kafka
-> Spark 聚合后写入 MySQL
-> Flask API 按登录用户 creator_id 读取数据
-> Vue 前端展示该用户自己的数据
```

核心原则：

```text
所有达人共用同一条 Flume/Kafka/Spark 链路。
不同达人之间靠 creator_id 隔离数据。
```

## 2. 当前已经实现的能力

- 支持账号登录。
- 支持新用户注册。
- 注册时创建独立 `creator_id`。
- 注册时选择模拟绑定平台：抖音、B站、小红书。
- 注册后自动初始化该达人的页面数据，避免新用户进入后空白。
- `mock_generator` 已支持从 MySQL 读取 `platform_accounts` 和 `videos`。
- `mock_generator --require-db` 模式下，如果数据库配置缺失或没有绑定数据，会直接失败，不再静默回退旧 mock。
- Flume 已能把生成器输出的 JSON 文件推入 Kafka。
- Spark Streaming 已能消费 Kafka 并通过 JDBC 写入 Windows 本机 MySQL。
- 已有一键脚本管理虚拟机侧三个进程：
  - Flume
  - mock_generator
  - Spark Streaming

虚拟机脚本路径：

```bash
/opt/creatorpulse/app/scripts/start_pipeline.sh
/opt/creatorpulse/app/scripts/status_pipeline.sh
/opt/creatorpulse/app/scripts/stop_pipeline.sh
```

本地项目脚本路径：

```bash
scripts/vm_pipeline/start_pipeline.sh
scripts/vm_pipeline/status_pipeline.sh
scripts/vm_pipeline/stop_pipeline.sh
```

## 3. 当前仍然存在的问题

### 3.1 当前是模拟绑定，不是真实平台授权

用户注册时选择“抖音 / B站 / 小红书”，目前只是写入模拟绑定配置：

```text
platform_accounts.binding_status = BOUND
```

这并不等于已经完成真实平台 OAuth 或 API 授权。

MVP 阶段这样是合理的，因为项目重点是打通数据链路。但后续如果要接真实平台，需要把“模拟绑定”和“真实授权”在产品文案、数据库字段和页面状态上区分清楚。

### 3.2 当前实时事件主要只有 video_stats

现在 `mock_generator` 主要生成：

```text
event_type = video_stats
```

所以 Spark 当前主要更新：

```text
spark_platform_metric_summaries
spark_video_follower_contributions
```

这意味着部分页面模块会有持续变化，但不是所有页面数据都会实时变化。

例如以下数据目前仍主要来自注册时初始化：

- 粉丝画像
- 热点话题
- 账号 7 天趋势
- insight
- 部分推荐动作

### 3.3 Spark 写入的是聚合结果表，不是完整替代业务表

当前 Spark Streaming 写入的是结果表：

```text
spark_platform_metric_summaries
spark_video_follower_contributions
```

它还没有持续写入或更新这些主业务表：

```text
video_metric_snapshots
creator_metric_snapshots
insights
```

所以现在系统更准确的定位是：

```text
Spark 提供部分实时分析结果。
基础页面数据仍由初始化数据和 MySQL 业务表提供。
```

### 3.4 一键脚本还不是系统服务

当前一键脚本可以启动后台进程，但它们不是 systemd 服务。

如果虚拟机重启，需要重新执行：

```bash
/opt/creatorpulse/app/scripts/start_pipeline.sh
```

后续可以升级为：

```text
systemd service
或 restart_pipeline.sh
或健康检查 + 自动重启机制
```

### 3.5 日志和错误处理仍比较原始

当前主要通过日志目录排查问题：

```bash
/opt/creatorpulse/data/logs
```

现阶段没有：

- 链路异常告警
- 错误事件表
- Spark 写入失败自动恢复说明
- 前端链路健康提示
- Kafka offset 可视化

这对开发阶段够用，但后续调试会变慢。

### 3.6 注册和测试会留下数据

注册接口会真实写入：

- `users`
- `creators`
- `platform_accounts`
- 初始化视频、粉丝、insight、Spark 结果数据

测试注册也会留下数据。短期可以接受，但后面需要补：

```text
测试达人清理脚本
或内部管理页面删除测试达人
```

否则数据库会积累很多无效账号和模拟数据。

### 3.7 多达人隔离依赖 creator_id 纪律

当前数据隔离逻辑是对的：

```text
登录用户 -> session -> user_id -> creator_id -> WHERE creator_id = ?
```

但是后续所有新增表、新接口、新 Spark 写入都必须遵守：

```text
业务数据必须带 creator_id。
查询必须按 creator_id 过滤。
```

否则会出现不同达人之间数据串读的问题。

## 4. 建议下一步

建议下一步优先做“链路健康与模拟控制台”，而不是继续堆分析页面。

建议页面：

```text
/admin/simulation
```

核心用途：

- 看当前有多少达人。
- 看每个达人绑定了哪些平台。
- 看 Flume 是否运行。
- 看 mock_generator 是否运行。
- 看 Spark Streaming 是否运行。
- 看 Kafka 最新 offset。
- 看 MySQL 最新写入批次。
- 看最近事件属于哪些 `creator_id`。
- 支持手动生成一批事件。
- 支持暂停或恢复某个达人/平台的模拟。

这会让后续调试效率明显提高。

## 5. 当前结论

当前主链路已经打通，MVP 可以继续往前走。

现在最大的问题不是“数据能不能流通”，而是：

```text
1. 哪些数据还没有实时化。
2. 链路状态缺少可视化。
3. 测试数据缺少清理能力。
4. 后续所有新增数据都必须严格带 creator_id。
```
