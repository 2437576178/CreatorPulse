## 1. 一条完整的模拟数据应该包含哪些内容？

### 1.1 核心问题：不同平台数据格式不同

抖音返回 `digg_count`，B站返回 `like`，小红书返回 `liked_count`，YouTube 返回 `likeCount`。同一个"点赞数"在不同平台有不同字段名。必须先归一化。

### 1.2 解决方案：统一标准化事件

Flume 采集后统一转换为以下标准格式推送到 Kafka：

### 1.3 五种事件类型

| event_type | Kafka Topic | 拉取频率 | 核心字段 |
|------------|-------------|:---:|----------|
| `video_stats` | `video_stats_topic` | 每 5 秒 | stats + growth + traffic_source + audience |
| `creator_stats` | `creator_stats_topic` | 每 30 秒 | follower + total_followers + net_growth |
| `comment` | `comment_topic` | 每 10 秒 | comment_text + sentiment + sentiment_score |
| `danmaku` | `danmaku_topic` | 每 5 秒 | danmaku_text + video_timestamp（B站特有） |
| `topic_trend` | `topic_trend_topic` | 每 60 秒 | hashtag + heat_score + platform_breakdown |

### 1.4 字段设计逻辑

| 分类 | 字段 | 为什么需要 |
|------|------|-----------|
| **标识** | `event_id` / `event_type` / `platform` / `fetch_time` | 事件唯一标识 + 平台区分 + 时序计算 |
| **创作者** | `creator_id` / `mcn_id` / `category` / `total_followers` | RT-04 粉丝统计、OF-05 粉丝画像 |
| **内容** | `content_id` / `content_type` / `publish_time` / `publish_hour` / `tags` | RT-05 新视频播放曲线、OF-03 内容类型分析、OF-04 发布时间分析 |
| **stats** | `play_count` / `like_count` / `comment_count` / `share_count` / `interaction_rate` | RT-01 ~ RT-03 全部播放和互动指标 |
| **growth** | `play_growth_5s` / `play_growth_1h` / `is_accelerating` / `velocity_score` | RT-01 实时变化、RT-08 异常检测（加速/减速） |
| **follower** | `total_followers` / `new_followers_today` / `net_growth` | RT-04 粉丝飞轮 |
| **traffic_source** | `recommend_ratio` / `search_ratio` / `follow_ratio` | OF-06 流量来源分析 |
| **audience** | `age_group_top` / `gender_ratio` / `top_region` | OF-05 粉丝画像聚合 |
| **monetization** | `estimated_revenue_cny` / `cpm_cny` | OF-12 平台 ROI 分析 |

## 2. 具体业务流程

### 2.1 七阶段业务流程全景

```
第一阶段：内容发布
  创作者在 6 个平台同时发布同一个视频
  抖音/B站/小红书/视频号/YouTube/微博 → 各自生成独立的 content_id

第二阶段：数据采集（Flume）
  Flume HTTP Source 定时轮询各平台 API（模拟阶段用 Shell 脚本代替）
  → 解析各平台不同格式的 JSON
  → 统一转换为 CreatorPulse 标准事件格式
  → 计算 growth（增量 = 当前值 - 上次快照值）
  → 推送到 Kafka 对应 Topic

第三阶段：消息缓冲（Kafka node1:9092）
  5 个 Topic × 多 Partition → 解耦采集和计算

第四阶段：实时分析（Spark Streaming）
  播放聚合器 / 互动率计算器 / 粉丝追踪器 / 热点检测器 / 竞品追踪器 / 异常检测器
  → 滑动窗口聚合 → 写入 MySQL 实时表

第五阶段：离线分析（SparkSQL）
  每天/每周定时执行：全平台日报、内容类型分析、发布时间分析、粉丝画像、竞品分析

第六阶段：智能推荐（Spark MLlib）
  选题推荐(ALS+时序预测) / 发布时间推荐(K-Means) / 标题优化(TF-IDF+回归)

第七阶段：API + 前端（Flask + Vue）
  创作者大屏：实时播放飞轮 / 互动面板 / 粉丝增长 / 热点雷达 / 竞品追踪 / 智能推荐
```

### 2.2 关键业务时序

```
T+0s    创作者点击"发布" → 视频同时在 6 个平台上线
T+5s    Flume 首次拉取到各平台数据 → 推送到 Kafka
T+10s   Spark Streaming 第一个窗口输出 → 实时播放飞轮开始转动
T+30s   互动率数据首次可显示 → 前端大屏更新
T+1min  新视频播放曲线开始绘制
T+5min  如果播放量加速增长 → 热点检测器标记 "可能爆款"
T+1h    播放增长趋于稳定 → growth.is_accelerating 变为 false
T+24h   发布满一天 → 离线分析生成次日报告
T+7d    离线分析生成周报，推荐引擎更新选题建议
```

## 3. 增长率计算：三层差分架构

### 3.1 核心原理：快照差分

增长率不是"数据自带的"，而是通过"对比前后两个快照的差值"计算出来的：

```
时间线 →
T+0s    第1次拉取 → play_count = 125,000  (基线)
T+5s    第2次拉取 → play_count = 126,500  (快照)
                      play_growth_5s = 126,500 - 125,000 = +1,500
T+10s   第3次拉取 → play_count = 128,000
                      play_growth_5s = 128,000 - 126,500 = +1,500
```

### 3.2 三层差分架构

| 层级 | 技术方案 | 粒度 | 用途 |
|:---:|----------|:---:|------|
| **Layer 1** | Spark Streaming `updateStateByKey` | 5s/30s | 实时播放飞轮、加速检测 |
| **Layer 2** | MySQL `snapshot_state` 表 + `ON DUPLICATE KEY UPDATE` | 1min/1h | 分钟级趋势、基线累计增长 |
| **Layer 3** | SparkSQL 查询历史表 | 24h/7d/30d | 周/月对比、环比/同比 |

### 3.3 MySQL 快照状态表

```sql
CREATE TABLE IF NOT EXISTS snapshot_state (
    content_id          VARCHAR(50)  NOT NULL,
    platform            VARCHAR(20)  NOT NULL,
    last_play_count     BIGINT DEFAULT 0     COMMENT '上次播放量（用于差分）',
    last_fetch_time     DATETIME DEFAULT NULL,
    baseline_play_count BIGINT DEFAULT 0     COMMENT '基线播放量',
    baseline_fetch_time DATETIME DEFAULT NULL,
    current_play_count  BIGINT DEFAULT 0     COMMENT '当前播放量',
    current_fetch_time  DATETIME DEFAULT NULL,
    update_time         DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (content_id, platform)
) ENGINE=InnoDB COMMENT='视频快照状态表（用于计算增长率）';
```

写入时用 `INSERT ... ON DUPLICATE KEY UPDATE` 原子操作：把当前值移到 `last_*` 位置，再写入新值，同时计算增长率写入 `realtime_growth_stats` 表。

## 4. 热点话题热度：三维增长模型

### 4.1 热度评分公式

```
话题热度 = Σ(各平台热度 × 平台权重)

单平台热度 = 参与视频数 × 0.4 + 搜索量 × 0.3 + 互动量 × 0.3

平台权重: 抖音 1.5 / 小红书 1.3 / B站 1.0 / 微博 1.2 / 视频号 0.8 / YouTube 0.7

热度增长判定:
  surging (爆发):   增长率 > 50%/h 且跨平台数 ≥ 3
  rising (上升):    增长率 15-50%/h
  stable (稳定):    增长率 -15% ~ +15%/h
  declining (下降):  增长率 < -15%/h
```

### 4.2 跨平台爆发检测

Spark Streaming 通过 `groupByKeyAndWindow` 在 120s 窗口内统计话题标签跨平台出现次数和热度变化率，当增长率 > 50% 且跨平台数 ≥ 3 时触发 `surging` 告警。

## 5. 真实数据获取：三种路径

### 5.1 各平台开放程度对比

| 平台 | 官方 API | 数据丰富度 | 是否需要创作者授权 | 难度 |
|------|:---:|:---:|:---:|:---:|
| YouTube | ✅ YouTube Data API v3 | ⭐⭐⭐⭐⭐ | OAuth 2.0 | 🟢 低 |
| B站 | ⚠️ 社区逆向 API | ⭐⭐⭐⭐ | Cookie | 🟡 中 |
| 抖音 | ✅ 开放平台 | ⭐⭐ 仅总量 | OAuth + 企业认证 | 🔴 高 |
| 小红书 | ❌ 无 | ⭐ | — | 🔴 极高 |
| 微博 | ✅ 微博开放平台 | ⭐⭐⭐ | OAuth | 🟡 中 |
| 视频号 | ❌ 无 | ⭐ | — | 🔴 极高 |

### 5.2 三种真实数据获取路径

| 路径 | 原理 | 适用平台 | 优劣势 |
|------|------|----------|--------|
| 🥇 **OAuth 授权** | 创作者授权后调官方 API | YouTube/抖音/微博 | 数据可靠，但抖音只返回总量 |
| 🥈 **浏览器插件** | Chrome 插件识别创作者后台数据 | B站/抖音创作平台 | 能拿到明细，但依赖创作者开电脑 |
| 🥉 **第三方服务** | 新榜/飞瓜/蝉妈妈 API | 小红书/视频号/竞品 | 最省事，但需要付费 |

## 6. 动态增长模拟：Shell 增长引擎

### 6.1 思路：静态基线 + 动态增长函数

Mock 数据文件不是存"播放量 128,500"这个固定值，而是存**增长参数**：

```csv
content_id,platform,initial_play,growth_type,peak_rate,decay_half_life,total_potential
BV001,bilibili,120000,viral,500,4,850000
DY001,douyin,0,normal,120,6,320000
XHS001,xiaohongshu,0,slow,30,12,120000
```

### 6.2 四种增长曲线

| growth_type | 数学特征 | 适用场景 |
|-------------|----------|----------|
| `viral` | 逻辑斯蒂 S 曲线：爆发增长 → 快速饱和 | 爆款视频 |
| `normal` | 前 4h 线性增长 → 后指数衰减趋近 | 正常视频 |
| `slow` | 对数曲线：缓慢但持续增长 | 教程/知识类长尾视频 |
| `reborn` | 先平缓 → 某时刻突增 → 再平缓 | 被算法二次推荐 |

### 6.3 Shell 脚本核心逻辑

```bash
# 计算视频已存在的秒数
age_seconds=$(( NOW - publish_ts ))

# 根据增长类型计算 growth_ratio（0.0 ~ 1.0）
case $growth_type in
    viral)
        # 逻辑斯蒂曲线: 1 / (1 + e^(-k*(t - midpoint)))
        growth_ratio=$(echo "1 / (1 + e(-$k * ($age_seconds - $midpoint)))" | bc -l)
        ;;
    normal)
        # 前4h线性 0→0.7，后指数衰减 0.7→1.0
        if [ $age_seconds -lt 14400 ]; then
            growth_ratio=$(echo "$age_seconds / 14400 * 0.7" | bc)
        else
            growth_ratio=$(echo "0.7 + 0.3 * (1 - e(-$decay * ($age_seconds - 14400)))" | bc -l)
        fi
        ;;
    slow)
        # 对数曲线
        growth_ratio=$(echo "l($age_seconds + 1) / l($age_seconds + 1000)" | bc -l)
        ;;
esac

# 计算当前累计播放量
current_play=$(echo "$initial_play + ($total_potential - $initial_play) * $growth_ratio" | bc)

# 计算 5 秒增量
play_5s_ago=$(echo "$initial_play + ($total_potential - $initial_play) * $growth_ratio_5s" | bc)
play_growth_5s=$(( current_play - play_5s_ago ))

# 输出 JSON → Flume 监控文件
echo "{...\"play_count\":${current_play},\"play_growth_5s\":${play_growth_5s}...}" >> output.log
```

### 6.4 效果：Flume 每次读到的数据都在涨

```
T=0s    → {"play_count": 120000, "play_growth_5s": 0}
T=5s    → {"play_count": 120150, "play_growth_5s": 150}
T=10s   → {"play_count": 120320, "play_growth_5s": 170}   ← 加速!
T=30s   → {"play_count": 120900, "play_growth_5s": 190}
T=1h    → {"play_count": 185000, "play_growth_5s": 850}
T=4h    → {"play_count": 720000, "play_growth_5s": 2500}   ← 爆发期
T=24h   → {"play_count": 920000, "play_growth_5s": 50}     ← 接近饱和
```

前端的实时播放飞轮每 5 秒跳一次，数值持续增长，就像真正的视频播放量一样。
