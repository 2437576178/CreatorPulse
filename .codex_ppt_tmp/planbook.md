# 项目背景

[]{#_Toc90390519 .anchor}随着抖音、B
站、小红书等短视频平台快速发展，大量自媒体创作者同时运营多个内容账号。各平台后台数据相互独立分散，创作者无法统一查看播放、点赞、粉丝增长、流量来源等完整数据，存在诸多业务痛点：

**1.数据割裂**：多平台数据无统一汇总窗口，跨平台对比效率极低；

**2.指标无统一标准**：各平台互动字段命名不统一，点赞、收藏等数据无法直接对比；

**3.无法实时监测内容增长**：视频爆发、流量暴跌、粉丝流失无实时预警；

创作缺少数据支撑：选题、发布时间、内容类型仅凭经验，无量化分析建议；

> **4.缺少完整大数据实践链路**：传统 Web
> 项目仅使用单数据库，无法学习采集、消息队列、流式计算完整流程。
>
> 基于以上痛点，本项目搭建一套**Flume+Kafka+Spark标准大数据实时处理平台**，模拟**多平台采集标准化数据**，通过**实时计算聚合指标**，搭配
> **Flask+Vue
> 搭建可视化分析后台**，实现达人全维度数据监测、内容复盘、智能创作推荐，同时完整覆盖大数据实训核心知识点。

# 项目介绍

## 项目概述

[]{#_Toc90390521 .anchor}**CreatorPulse
是面向自媒体创作者的端到端实时大数据分析系统**。系统分为两套可切换运行模式：

**大数据正式链路（flume_kafka_spark）**：模拟平台生成标准化事件文件 →
Flume 采集 → Kafka 消息缓冲 → Spark Streaming 实时聚合计算 → MySQL
持久化 → 前后端可视化展示；

**简易调试链路（direct_mysql）**：跳过大数据组件，模拟脚本直接写入
MySQL，用于前端快速调试、现场演示兜底。

系统统一抖音、B
站、小红书等平台数据字段，标准化播放、互动、粉丝、流量指标，提供 6
大可视化分析页面，同时内置规则引擎生成可解释的创作诊断、风险提醒、行动建议，完整实现数据从产生、传输、计算、存储到可视化全生命周期流程。

# 2.项目功能

![](media/image3.png){width="5.488194444444445in"
height="3.1777777777777776in"}

**一.底层功能**:

**标准化模拟事件生成**: 自动模拟多平台视频统计事件，基于 S
型爆发、对数长尾增长曲线动态更新播放、涨粉增量；采用临时文件 +
重命名机制，保证文件读写完整性，记录状态文件实现重启续增长。

> **Flume 文件采集**: 监听指定目录 JSON 事件文件，过滤临时 tmp
> 文件，批量推送数据至对应 Kafka 主题，实现稳定数据采集。
>
> **Kafka 消息队列管理**: 划分 5 类独立业务
> Topic，实现采集与计算解耦，削峰填谷；提供生产、消费、连通性测试脚本。
>
> **Spark 实时流式计算**: 消费 Kafka 事件，统一字段、实时聚合播放 / 互动
> / 粉丝指标，计算转粉率、粘性指数、账号健康度，批量写入 MySQL
> 实时结果表。
>
> **Spark 离线批处理**: 按日 / 周 /
> 月汇总历史原始事件，生成日报、周报、月报，支持历史数据批量重算。
>
> **全链路容错与回退**: Spark 开启检查点恢复
> offset；链路故障一键切换直写 MySQL 模式，保证演示可用。

2.  **后端服务功能:**

    · **用户注册、登录、多平台账号模拟绑定；**

**· 双数据源适配（Mock 本地 JSON / MySQL 数据库）；**

**· 统一 API 视图模型，标准化接口返回数据；**

**· 规则洞察引擎，自动生成诊断、机会、风险、行动建议；**

**· 全链路预检工具，提前检测 MySQL、Kafka、Spark 连接故障；**

**· 离线报表查询、历史数据重算请求管理。**

3.  **前端可视化功能:**

·
**增长总览页面**：账号健康度、实时粉丝趋势、播放转粉漏斗、热门视频排行；

**· 粉丝分析页面：粉丝净增趋势、各平台涨粉贡献、粉丝画像（年龄 / 性别 /
地域 / 活跃时段）；**

**·
视频分析页面：视频列表、单视频增长曲线、涨粉贡献排行、评论情感分析；**

**·
内容分布页面：平台投入产出、内容类型效率、最佳发布时段、流量来源占比；**

**· 机会建议页面：热点话题适配、选题 / 发布时间 /
标题优化、平台运营建议；**

**· 个人中心页面：账号设置、平台绑定、日报周报月报查看、链路状态监控。**

**大数据数据流流程图（核心技术链路）:**部署于 Linux 虚拟机大数据集群与
Windows 本机服务两大环境。由 Mock 工具模拟新媒体用户行为数据，经 Flume
采集后送入划分五类业务主题的 Kafka 集群，再由 Spark Streaming
完成实时数据处理，分别开展实时指标统计、原始事件归档，依托 Spark
离线任务生成多周期运营报表，通过 JDBC 跨主机写入 MySQL 数据库。最后由
Flask 后端读取数据，经由 Vue
前端实现数据可视化，完成从数据采集、计算存储到业务分析展示的全链路大数据业务流程。

##  数据库核心表结构设计

### 档案基础表（一次性初始化）

1\. users：登录账号、用户密码、关联 creator_id；

2.creators：达人昵称、领域标签、时区；

3.platform_accounts：达人绑定平台、同步状态；

4.videos：视频标题、发布时间、内容类型基础信息。

### 实时指标快照表（Spark 实时写入）

1.video_metric_snapshots：单视频播放、点赞、新增粉丝、转粉率；

2.creator_metric_snapshots：账号总播放、净增粉丝、账号健康度、粘性指数；

3.spark_platform_metric_summaries：各平台汇总指标；

4.spark_video_follower_contributions：视频涨粉贡献排行；

5.video_traffic_source_metrics：视频各流量来源数据。

### 离线归档 & 报表表

1.raw_video_stat_events：原始事件归档，用于历史重算；

2.offline_creator_daily_metrics：达人日汇总指标；

3.creator_reports：日报、周报、月报存储表。

### 洞察分析表

1.insights：诊断、机会、风险结论；

2.insight_evidence_metrics：结论对应的指标证据；

3.recommended_actions：系统给出创作行动建议。

# 实现技术

**分层整体技术栈：**

![](media/image9.png){width="5.405555555555556in"
height="2.6347222222222224in"}

### **各模块技术实现说明：**

### 一. 数据采集模块（Flume）

- 1.Source：spooldir监控指定目录，正则匹配.json有效事件忽略.tmp
  临时文件；

- 2.Channel：memory 内存通道，缓存 10000 条消息应对流量峰值；

  3.Sink：Kafka 专用输出，批量 100 条发送，acks=1 保证基础可靠性；

- 4.配置文件独立管理，单独脚本启停采集服务。

### 二. 消息队列模块（Kafka）

1.按事件类型拆分 5 个独立 Topic，每个 Topic 设置 3
分区提升并行消费能力；

2.配套工具脚本：消息生产、消费、Windows 跨虚拟机连通性检测；

支持控制台实时查看事件，方便链路调试。

### 三. Spark 计算模块

1.Spark Streaming：5 秒滑动窗口消费
Kafka，使用状态管理持续累计视频播放、粉丝增量；统一多平台字段，计算转粉率、粘性指数、账号健康度，JDBC
跨主机写入 Windows MySQL；开启 checkpoint 实现宕机恢复；

2.Spark SQL 离线批处理：读取原始归档事件，按天 /
周聚合指标，自动生成日报周报；

3.Spark MLlib：基于聚类、时序规则实现发布时间、选题推荐逻辑。

### 四. MySQL 存储模块

1.分档案、实时指标、离线归档、洞察四类数据表，按 creator_id
隔离多达人数据；

2.JDBC 连接 Spark，PyMySQL 供 Flask 读取；

3.提供建表、Mock 批量导入脚本，一键初始化数据库。

### 五. Flask 后端模块

1.数据源工厂模式：环境变量切换 mock/json/mysql 双数据源，前端无感知；

2.OpenAPI 标准化接口，6 大页面统一 ViewModel；

3.规则引擎：基于指标数值自动生成 Insight 洞察，附带证据与行动建议；

4.预检脚本：检测 MySQL、Kafka、Spark 连接，提前拦截配置错误。

### 六. Vue 前端模块

1.组件化封装指标卡片、诊断条、图表、行动建议模块；

2.六个一级页面，深色数据大屏风格；

3.定时轮询接口，10 秒刷新实时增长数据；

4.页面数据全部由 API 提供，无硬编码静态数字。

### 7. 运维与容错技术

1.虚拟机 Shell 脚本：一键启动 / 停止 / 查看 Flume、Mock、Spark
三大进程；

2.双运行模式切换，大数据组件故障可回退直写 MySQL；

3.所有账号、地址、密码存放.env 配置文件，不提交代码仓库。

# 项目开发流程与实施阶段

1.  **阶段 1：统一 Mock 数据与规则引擎**
    搭建标准化事件结构，定义指标计算公式、洞察判断规则，产出统一模拟数据集，保证数据口径统一。

2.  **阶段 2：Flask 后端 API 开发** 设计页面视图模型，编写 6 大页面 REST
    接口，生成 OpenAPI 接口文档，实现 Mock 数据源读取。

3.  **阶段 3：Vue 前端工程开发**
    迁移全部页面，封装通用可视化组件，对接后端 Mock 接口，完成页面交互。

4.  **阶段 4：MySQL 数据库设计**
    设计全套业务表，编写建表、数据导入脚本，实现持久化存储。

5.  **阶段 5：后端适配 MySQL 数据源** 开发数据库仓库层，工厂模式切换
    Mock/MySQL，两套接口输出完全一致。

6.  **阶段 6：Spark 静态计算验证** 不接入 Kafka，本地读取 Mock
    文件，验证 Spark 通过 JDBC 写入 Windows MySQL 可行性。

7.  **阶段 7：虚拟机 Kafka 连通调试** 调试 Windows
    与虚拟机网络、防火墙、Kafka 监听地址，验证消息生产消费。

8.  **阶段 8：完整大数据链路打通** Mock 生成→Flume 采集→Kafka 缓冲→Spark
    Streaming 实时计算→MySQL→前后端完整闭环，全功能验收测试。

# **六．**总结

# 技术收获

本项目完整落地**Flume+Kafka+Spark**标准大数据实时处理架构，同时结合 Web
前后端开发，覆盖数据采集、消息中间件、流式计算、离线报表、可视化全流程。团队掌握跨主机服务通信、多数据源适配、指标标准化、流式容错、分层架构等工程化开发思想，解决了
Windows 与虚拟机跨端通信、JDBC
远程写入、文件采集完整性等实操难点，完整贴合大数据实训课程核心要求。

## 业务收获

团队梳理自媒体平台完整数据指标体系，搭建从原始事件到可视化结论的完整业务链路，理解
"原始数据 - 聚合指标 - 智能洞察"
分层设计思路，所有分析结论均具备可量化数据支撑，脱离纯静态演示页面，实现动态实时数据驱动系统。

##  现存不足与优化方向

1.  当前仅模拟平台数据，未对接抖音、B 站官方真实开放 API；

2.  评论、弹幕、热点 Topic 完整实时链路待扩展开发；

3.  缺少可视化运维监控后台，链路状态依赖日志查看；

4.  海量原始事件未接入分布式存储 HDFS，长期存储存在性能瓶颈；

5.  智能推荐仅使用基础规则，未引入深度学习模型做深度预测。
